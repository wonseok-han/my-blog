# Self-managed GitLab CE에서 Jira Cloud 연동하기

사내에서 방화벽까지 다 막힌 Self-managed GitLab CE(~~EE 좀 사주지...~~)를 사용하면서 Jira Cloud와의 연동 필요성을 느꼈다.

기존에는 Jira에서 이슈를 만들고, GitLab에서 커밋하고, 다시 Jira에 돌아와서 수동으로 상태를 바꾸는 식으로 작업했다. 이슈 번호를 커밋 메시지에 적어도 서로 연결되지 않으니, "이 커밋이 어떤 이슈랑 관련됐지?"를 추적하려면 양쪽을 번갈아 확인해야 했다.

### 연동하면 뭐가 좋은가?

| 기능 | 설명 |
|------|------|
| **자동 링크** | 커밋/MR 메시지에 이슈 키(`PROJ-123`)만 적으면 Jira 이슈에 자동으로 링크가 생성된다 |
| **상태 자동 변경** | `#close`, `#done` 같은 Smart Commit 명령어로 커밋과 동시에 Jira 이슈 상태를 변경할 수 있다 |
| **개발 추적** | Jira 이슈에서 관련 커밋, MR, 브랜치를 한눈에 확인할 수 있다 |
| **컨텍스트 전환 감소** | GitLab ↔ Jira를 오가며 수동으로 업데이트할 필요가 없어진다 |

이런 이유로 연동을 시도했고, 처음에는 Atlassian Marketplace의 공식 앱인 **GitLab for Jira Cloud**를 통해 시도했으나 GitLab CE 버전의 제약으로 실패했다. 최종적으로 **Jira Issues Integration** 방식으로 연동에 성공했다.

이 글에서는 두 가지 연동 방식의 설정 과정과 트러블슈팅 경험을 공유한다.

* **시도 환경**: GitLab Community Edition (CE) v18.5.1 / Jira Cloud (Atlassian Cloud)

---

## 방법 1: GitLab for Jira Cloud 앱 (실패)

Atlassian Marketplace에서 제공하는 공식 앱을 통한 연동 방식이다. 앱이 GitLab과 Jira 사이의 개발 정보(커밋, MR, 브랜치, 파이프라인 등)를 양방향으로 동기화해 준다.

### 필수 권한

* **Jira Cloud 측**: 조직 관리자 (Org Admin) — `org-admins` 또는 `site-admins` 그룹 소속
    
    → Gemini님께서 `site-admins` 권한으로도 된다고 했는데 난 아무리해도 안되서 `org-admins`를 받아서 했다.

* **GitLab 측**: 인스턴스 관리자 (Admin Area) — OAuth Application 등록 및 시스템 설정용

### 필수 요구사항

| 항목 | 요구사항 |
|------|----------|
| **GitLab 버전** | 15.7 이상 (Marketplace 설치 기준) |
| **HTTPS** | 공인 인증서 또는 전체 체인 인증서 포함 필수 |
| **공개 접근** | 인스턴스가 인터넷에서 직접 접근 가능해야 함 |

### 연동 절차

#### Step 1. Jira Cloud에서 앱 설치

1. Jira > **앱** > **더 많은 앱 살펴보기** 이동
2. **"GitLab for Jira Cloud"** 검색 후 **"Get it now"** 선택

#### Step 2. 네트워크 설정 (방화벽)

| 방향 | 대상 | 참고 |
|------|------|------|
| **인바운드** | 사내 GitLab 서버로 들어오는 GitLab IP 대역 | [GitLab IP Range](https://docs.gitlab.com/user/gitlab_com/#ip-range) |
| **아웃바운드** | GitLab 서버에서 Jira Cloud로 나가는 Jira IP 대역 | [Atlassian IP Range](https://support.atlassian.com/organization-administration/docs/ip-addresses-and-domains-for-atlassian-cloud-products/#Outgoing-Connections) |

Marketplace 설치 시 추가로 gitlab.com으로의 아웃바운드 연결도 필요하다.

#### Step 3. GitLab에서 OAuth Application 생성

GitLab Admin Area에서 OAuth 애플리케이션을 등록한다.

1. 좌측 하단 **Admin** 클릭
2. **Applications** > **New application** 선택
3. 아래와 같이 설정:

| 항목 | 값 |
|------|-----|
| **Name** | `Jira Connect` (또는 식별 가능한 이름) |
| **Redirect URI** | Marketplace 설치: `https://gitlab.com/-/jira_connect/oauth_callbacks` |
|                  | 수동 설치: `<인스턴스URL>/-/jira_connect/oauth_callbacks` |
| **Trusted** | ☐ 해제 |
| **Confidential** | ☐ 해제 |
| **Scopes** | ✅ `api` (이것만 선택) |

4. **Save application** 클릭
5. 생성된 **Application ID** 값을 복사해 둠

> `api` 스코프는 전체 읽기/쓰기 권한을 부여하지만, 실제로 앱은 그룹 표시 및 그룹 연결에만 사용한다. Confidential을 해제해야 로그인 오류를 방지할 수 있다.

#### Step 4. GitLab Admin Settings에서 Jira 앱 구성

1. **Admin** > **Settings** > **General** 이동
2. **GitLab for Jira App** 섹션 펼치기
3. 설치 방식에 따라 설정:

**Marketplace 설치 시:**

| 항목 | 값 |
|------|-----|
| **Jira Connect Proxy URL** | `https://gitlab.com` |
| **Jira Connect Application ID** | Step 3에서 복사한 Application ID |

**수동 설치 시:**

| 항목 | 값 |
|------|-----|
| **Jira Connect Proxy URL** | 비워둠 (공란) |

4. **Save changes** 클릭

#### Step 5. Jira Cloud에서 Self-managed 인스턴스 연결

1. Jira에서 GitLab for Jira 앱 열기
2. 연결 유형 선택 화면에서 **"GitLab (self-managed)"** 선택
   * `GitLab.com (SaaS)` — SaaS 사용자용 (Most common)
   * `GitLab (self-managed)` — 우리가 선택해야 하는 항목
3. 아래 4개 체크항목을 모두 확인 후 체크:
   * ✅ **Prerequisites** — 필수 요구사항 충족 (HTTPS, 공개 접근, GitLab 버전 등)
   * ✅ **Set up OAuth authentication** — Step 3에서 OAuth Application 생성 완료
   * ✅ **Set up your instance** — Step 4에서 Admin Settings 구성 완료
   * ✅ **Jira user requirements** — Jira 사용자가 `org-admins` 또는 `site-admins` 그룹에 속해야 함
4. 사내 GitLab 도메인 입력 (예: `https://gitlab.example.com`)
5. GitLab 로그인 및 권한 승인

### 배경: DVCS 커넥터는 왜 안 되나?

여담이지만, 예전에는 Jira DVCS 커넥터라는 방식으로도 GitLab과 Jira Cloud를 연동할 수 있었다. 하지만 [공식 문서](https://docs.gitlab.com/integration/jira/dvcs/)에 따르면 Jira Cloud용 DVCS 커넥터는 **GitLab 16.0에서 제거**되었다. 현재 v18.5.1에서는 선택지 자체가 없고, 반드시 **GitLab for Jira Cloud 앱**을 사용해야 한다.

### 장애 요인 분석

여기까지 진행했으나, 최종 인증 단계에서 실패했다. 원인 분석 과정은 다음과 같다.

#### 장애 1: 네트워크 문제인가?

* **현상:** 인/아웃바운드 IP 대역 설정 후에도 인증 실패
* **검증:** 원인 파악을 위해 일시적으로 **Any Open**을 실시했으나, 여전히 동일한 에러(`401 Unauthorized` / `422 Unprocessable Content`)가 반복됨
* **결론:** 방화벽(네트워크)은 정상적으로 뚫려 있으나, 상위 계층인 **애플리케이션 인증 단계**에서 문제가 발생함을 확정

#### 장애 2: Self-managed CE 환경과 앱 간의 OAuth 인증 실패

* **현상:** 모든 권한과 네트워크 환경을 갖췄음에도 최종 인증 단계에서 거부됨
* **원인:** [공식 문서](https://docs.gitlab.com/administration/settings/jira_cloud_app/)에 따르면, GitLab for Jira Cloud 앱은 OAuth 2.0 인증 및 특정 API 조건을 충족해야 한다. 공식적으로는 Free 등급도 지원한다고 되어 있으나, 실제 Self-managed CE 환경에서는 앱이 요구하는 OAuth 2.0 핸드셰이크 규격이 CE 버전의 API 엔드포인트와 호환되지 않았다.
* **결과:** CE 서버가 Jira의 인증 요청을 '처리할 수 없는 요청'으로 간주하여 `401 Unauthorized` / `422 Unprocessable Content` 응답을 반환

### 결론

네트워크 팀과 협의하여 인/아웃바운드 IP 대역을 모두 개방하고 Any Open 테스트까지 마쳤으나 동일한 에러(401/422)가 발생했다. 이는 **망의 문제가 아닌 SaaS형 앱과 방화벽 다 막아놓은 Self-managed CE 버전 간의 아키텍처 충돌**로 판단된다. 대안을 찾아야 했다.

## 방법 2: Jira Issues Integration (성공)

공식 앱이 막혔으니, GitLab에 내장된 **Jira Issues Integration** 기능을 사용하는 방향으로 전환했다. 이 방식은 GitLab에서 Jira Cloud API를 직접 호출하는 구조라 CE에서도 문제없이 동작한다.

* **방식**: GitLab(발신) → Jira Cloud API(수신) 호출
* **장점**: 사내망 방화벽 Inbound 설정이 불필요하며, 프로젝트 단위로 즉시 설정 가능
* **주요 기능**:
    * 커밋/MR 메시지 내 이슈 키(`PROJ-123`) 자동 링크
    * Jira 이슈 내 GitLab 활동 웹링크 생성
    * 커밋 메시지를 통한 Jira 이슈 상태 자동 변경 (Transition)
* **단점**: 이슈 키를 포함한 커밋이 쌓일수록 Jira 이슈에 웹링크가 무분별하게 생성되어 다소 지저분해질 수 있다. 개발 패널처럼 커밋/MR을 깔끔하게 그룹핑해 주는 것이 아니라, 건건이 웹링크로 나열되는 방식이기 때문이다.

### Step 1. Jira API 토큰 발급

GitLab이 Jira에 데이터를 보낼 때 사용할 인증 수단을 먼저 생성한다.

1. [Atlassian 보안 설정](https://id.atlassian.com/manage-profile/security/api-tokens) 접속
2. **[API 토큰 만들기]** 클릭 후 레이블 입력 (예: `GitLab_Integration`)
3. 생성된 토큰을 복사하여 안전한 곳에 보관 (재확인이 불가능하므로 반드시 즉시 복사)

### Step 2. GitLab 프로젝트에서 Integration 설정

관리자 권한이 있는 GitLab 프로젝트에서 다음 설정을 진행한다.

1. 연동할 프로젝트의 왼쪽 사이드바에서 **Settings** > **Integrations** 클릭
2. 목록에서 **Jira Issues** 항목을 찾아 클릭
3. 아래와 같이 입력:

| 항목 | 입력 값 |
|------|---------|
| **Enable integration** | Active |
| **Trigger** | Commit, Merge request 체크 |
| **Web URL** | `https://[your-domain].atlassian.net` |
| **Jira API URL** | (공란 유지 - Web URL과 동일할 경우) |
| **Username** | Jira 로그인 이메일 주소 |
| **API token** | Step 1에서 발급받은 토큰 값 |

4. **[Save changes]** 를 눌러 최종 저장

### Step 3. 사용 방법 (Smart Commits)

연동 완료 후, 커밋 메시지에 Jira 이슈 키를 포함하면 자동으로 연동된다.

**단순 링크 생성:**

```
PROJ-123: 로그인 오류 수정
```

**이슈 상태 변경 (Transition):**

커밋과 동시에 Jira 이슈를 '완료' 등으로 변경하고 싶을 때 사용한다. (Jira 워크플로우 설정 필요)

```
PROJ-123: 결제 모듈 추가 #close
```

주요 명령어: `#close`, `#done`, `#comment`

### 연동 결과

설정을 마치고 이슈 키가 포함된 커밋이나 MR을 push하면, Jira 이슈의 **웹 링크** 섹션에 아래와 같이 자동으로 링크가 생성된다.

## 마치며

GitLab CE 환경에서 Jira Cloud 연동을 시도한다면, **GitLab for Jira Cloud 앱은 CE에서 연동이 굉장히 어렵다**는 점을 먼저 인지하는 게 중요하다. GitLab 16.0 이후 DVCS 커넥터도 제거되어 선택지가 이 앱뿐인데, 네트워크 설정이나 OAuth 구성까지 다 마쳐도 마지막 인증 단계에서 막히게 된다.

대신 **Jira Issues Integration**은 CE에서도 잘 동작하며, 커밋/MR과 Jira 이슈 간 자동 링크 및 상태 변경까지 지원한다. 양방향 동기화(개발 패널)까지는 안 되지만, 실무에서 필요한 핵심 기능은 충분히 커버된다. 다만 커밋이 쌓일수록 Jira 이슈에 웹링크가 건건이 나열되어 지저분해질 수 있다는 점은 감안해야 한다.

| 비교 | GitLab for Jira Cloud 앱 | Jira Issues Integration |
|------|--------------------------|------------------------|
| **CE 지원** | ❌ | ✅ |
| **설정 범위** | 인스턴스 전체 | 프로젝트 단위 |
| **방화벽 설정** | 인바운드 + 아웃바운드 | 아웃바운드만 |
| **커밋/MR 링크** | ✅ (개발 패널에 그룹핑) | ✅ (웹링크 개별 나열) |
| **Jira 개발 패널** | ✅ | ❌ |
| **Smart Commits** | ✅ | ✅ |
| **이슈 정리** | 깔끔 | 커밋이 많으면 웹링크 과다 |
